# Backup & Restore Baseline (PostgreSQL)

Минимально достаточный эксплуатационный baseline для резервного копирования и восстановления БД HLS Monitoring.

## Что бэкапится

- **Объект**: одна база PostgreSQL (`POSTGRES_DB`, по умолчанию `hls_monitoring`).
- **Метод**: `pg_dump` в формате plain SQL, сжатие gzip.
- **Включено**: схема и данные всех таблиц (пользователи, компании, проекты, стримы, check_jobs, check_results, алерты, инциденты, настройки Telegram и т.д.).

## Скрипты

| Скрипт | Назначение |
|--------|------------|
| `scripts/backup-db.sh` | Создаёт дамп в `BACKUP_DIR`, по умолчанию `./backups`. Удаляет бэкапы старше `RETENTION_DAYS`. |
| `scripts/restore-db.sh <file.sql.gz>` | Останавливает api/worker/frontend, восстанавливает БД из дампа, запускает сервисы. |

На проде используйте те же переменные окружения, что и для деплоя: передайте `ENV_FILE` и при необходимости `COMPOSE_FILE`, чтобы скрипты работали с вашим prod-стеком (см. ниже).

## Как часто

- **Рекомендация**: не реже одного полного бэкапа в сутки.
- Пример: cron в 03:30 UTC (см. `docs/deploy_timeweb.md`, раздел 8).

## Где хранить

- **Минимум**: отдельная директория на том же сервере (например `./backups` или `/opt/hls-monitoring/backups`), не в репозитории.
- **Лучше**: копирование на другой хост или в объектное хранилище (rsync, S3-совместимое, бакет облака) по расписанию после создания бэкапа.
- Файлы содержат чувствительные данные; права доступа и сеть должны ограничивать доступ.

## Переменные окружения

- `BACKUP_DIR` — каталог для файлов бэкапа (по умолчанию `./backups`).
- `RETENTION_DAYS` — удалять бэкапы старше N дней (по умолчанию 14).
- `ENV_FILE` — путь к env-файлу (например `.env.prod`). Если задан, скрипты подставляют его в `docker compose`.
- `COMPOSE_FILE` — опционально. На проде обычно не задаётся: скрипты сами подставляют `docker-compose.yml` и `docker-compose.prod.yml` при заданном `ENV_FILE`.

Для совпадения с prod-стеком на сервере запускайте так:

```bash
export ENV_FILE=.env.prod
BACKUP_DIR=/opt/hls-monitoring/backups RETENTION_DAYS=14 ./scripts/backup-db.sh
```

Перед первым запуском убедитесь, что в `.env.prod` заданы `POSTGRES_USER`, `POSTGRES_DB` (и при необходимости `POSTGRES_PASSWORD` через env, если compose его не подхватывает в контейнер — для `pg_dump`/`psql` внутри контейнера пароль берётся из окружения контейнера postgres).

## Restore (восстановление)

1. Убедиться, что выбран правильный файл бэкапа и что он соответствует окружению (prod/dev).
2. Остановить приложение (api, worker, frontend) — скрипт это делает сам.
3. Выполнить восстановление:
   ```bash
   export ENV_FILE=.env.prod   # на проде
   ./scripts/restore-db.sh ./backups/pg_20250106T033000Z.sql.gz
   ```
4. Проверить:
   - `GET /api/v1/health` и `GET /api/v1/ready` возвращают 200;
   - логин в UI, список стримов, один просмотр стрима.

**Внимание**: restore перезаписывает текущее состояние БД. Все данные, созданные после момента бэкапа, будут потеряны. На проде перед restore обязательно делайте дополнительный бэкап текущего состояния (если нужно сохранить возможность откатиться).

## Restore drill (проверка восстановления)

Периодически проверяйте, что бэкап действительно восстанавливается.

1. **Окружение**: лучше отдельный стенд или копия prod (например временный compose с другой БД).
2. **Шаги**:
   - Создать тестовую БД или использовать клон.
   - Запустить restore из последнего прод-бэкапа (или копии файла) в эту БД.
   - Убедиться, что `psql` завершился без ошибок.
   - Запустить api/worker против этой БД и проверить health/ready и базовые сценарии UI.
3. **Частота**: например раз в квартал или после смены процесса бэкапов.

Чеклист для drill:

- [ ] Выбран актуальный файл бэкапа.
- [ ] Восстановление выполнено без ошибок (код выхода 0).
- [ ] API health/ready — 200.
- [ ] Логин и просмотр стрима работают.
- [ ] Результаты проверок (check_results) видны в UI (если есть в бэкапе).
