# Rollback Runbook

Честный путь отката неудачного релиза без фальшивой автоматизации.

## Когда делать rollback

- После деплоя: health/ready падают или ключевые сценарии (логин, стримы) не работают.
- Обнаружена регрессия в проде, которую быстрее откатить, чем чинить вперёд.

## Что откатываем

1. **Приложение** — код и образы (через git commit и пересборку контейнеров).
2. **Конфигурация** — `.env.prod` (вручную или из своей копии).
3. **Миграции БД** — отдельно; откат схемы только при крайней необходимости и только с проверенным бэкапом.

## Rollback приложения

Используются скрипты `scripts/deploy.sh` (деплой с записью коммита для отката) и `scripts/rollback.sh` (откат на предыдущий коммит).

### Предусловия

- Перед каждым деплоем `scripts/deploy.sh` записывает текущий коммит в `.deploy_prev_commit`.
- Для отката нужен либо этот файл, либо известный стабильный коммит/тег.

### Шаги отката приложения

1. Подключиться к прод-серверу, перейти в каталог приложения (например `/opt/hls-monitoring`).

2. Проверить, какой коммит сейчас развёрнут и какой был до последнего деплоя:
   ```bash
   cat .deploy_current_commit   # текущий (плохой)
   cat .deploy_prev_commit      # предыдущий (целевой для rollback)
   ```

3. Запустить откат на предыдущий коммит:
   ```bash
   export ENV_FILE=.env.prod
   ./scripts/rollback.sh
   ```
   Или на конкретный коммит/тег:
   ```bash
   ./scripts/rollback.sh <commit-hash-or-tag>
   ```

4. Скрипт сделает:
   - `git fetch --all --prune`
   - `git checkout --detach <target-commit>`
   - запись целевого коммита в `.deploy_current_commit`
   - `docker compose --env-file .env.prod -f docker-compose.yml -f docker-compose.prod.yml up -d --build --remove-orphans`
   - вывод статуса контейнеров

5. Проверить:
   - `curl -fsS http://127.0.0.1:8080/api/v1/health`
   - `curl -fsS http://127.0.0.1:8080/api/v1/ready`
   - Логин в UI, список стримов, один стрим.

6. При необходимости вернуться на ветку (например `master`):
   ```bash
   git checkout master
   ```

### Rollback checklist (приложение)

- [ ] Определён целевой коммит (из `.deploy_prev_commit` или известный стабильный).
- [ ] Выполнен `./scripts/rollback.sh [commit]` с корректным `ENV_FILE`.
- [ ] Контейнеры подняты, `docker compose ps` без перезапусков.
- [ ] Health и ready возвращают 200.
- [ ] Проверены базовые сценарии в UI.
- [ ] Зафиксированы коммит до и после отката для отчёта.

## Rollback конфигурации

Если проблема из-за изменений в `.env.prod`:

1. Восстановить предыдущую версию файла из своей копии/бэкапа (не из git, если там секреты не хранятся).
2. Перезапустить стек:
   ```bash
   docker compose --env-file .env.prod -f docker-compose.yml -f docker-compose.prod.yml up -d
   ```
3. Проверить health/ready и UI.

Секреты и пароли в `.env.prod` не должны быть в репозитории; храните конфиг под контролем версий отдельно (например приватный репозиторий или секрет-хранилище) или ведите ручной changelog изменений.

## Миграции БД: что делать при откате

- **Текущая практика**: миграции только вперёд (up). Отдельных down-миграций в репозитории может не быть.
- **Если откатили код на коммит со старой схемой**:
  - Новая схема уже применена в БД — приложение старой версии может падать из-за несовпадения таблиц/колонок.
  - Варианты:
    1. **Предпочтительно**: снова выкатить последнюю версию кода и чинить баг (или отключить фичу), не откатывая схему.
    2. **Крайний случай**: откат схемы через скрипт отката миграций (`scripts/rollback_migrations.ps1` или аналог) **только** после создания и проверки полного бэкапа БД и только если вы точно знаете порядок отката миграций.

**Правило**: не запускать деструктивные SQL и откат миграций без актуального бэкапа и без документирования версий до/после.
