# SLO / SLI Baseline

Небольшой набор реально измеримых ориентиров для HLS Monitoring. Без enterprise-матрицы — только то, что можно наблюдать и использовать уже сейчас.

## SLI (Service Level Indicators)

Определены по данным, которые система уже отдаёт или может отдавать.

| SLI | Что измеряем | Как измерять |
|-----|----------------|--------------|
| **API availability** | Доля успешных проверок готовности API к приёму трафика | Доля ответов `GET /api/v1/ready` с HTTP 200 за окно (например 1 мин или 5 мин). Успех = 200, иначе (5xx, таймаут, нет ответа) = failure. |
| **API latency (ready)** | Время ответа готовности | Latency перцентили (p95/p99) для `GET /api/v1/ready` в пределах окна. Зависит от БД и пула. |
| **Check execution freshness** | Актуальность проверок стримов | Время с момента последнего успешного завершения check job по стриму до «сейчас». Можно аппроксимировать по данным `check_results` и метрикам worker. |
| **Worker processing health** | Доля успешных циклов воркера | Доля `worker_cycle_total{result="ok"}` от общего числа циклов за окно. Или обратная метрика: доля/темп `result="error"`. |
| **Alert delivery (если включены алерты)** | Успешная отправка в Telegram при срабатывании алерта | По логам worker или по счётчикам отправок/ошибок, если будут добавлены. Пока — качественный критерий: алерты приходят в чат при падении стрима. |

## SLO (Service Level Objectives)

Разумные целевые значения для текущего масштаба.

| SLO | Формулировка | Окно |
|-----|----------------|------|
| **API availability** | Не менее 99% успешных ответов ready за месяц (успех = 200 в течение таймаута, например 5 с). | 30 дней (или скользящее 30d). |
| **API ready latency** | p95 latency `GET /api/v1/ready` ≤ 2 с. | Скользящее 5 мин или 1 ч. |
| **Check freshness** | Для активных стримов с включёнными проверками: последний успешный результат проверки не старше 2× интервала опроса (например при 30 мин интервале — не старше 60 мин). | По данным за последние 24 ч. |
| **Worker health** | Не менее 95% циклов воркера в состоянии ok за сутки. Или: темп `worker_cycle_total{result="error"}` не превышает 5% от общего темпа циклов. | 24 ч. |
| **Alert delivery** | При включённых Telegram-алертах: доставка в течение 5 мин от момента фиксации инцидента (оценочно по логам). | По инцидентам. |

## Связь с наблюдаемостью

- **API ready** — напрямую из мониторинга (http probe к `/api/v1/ready`). Настройте алерт при двух подряд неуспехах или при breach SLO за окно.
- **Worker** — метрики `worker_cycle_total`, `worker_job_finalized_total` с портами worker (см. `docs/metrics_health_alerting.md`). Алерт при росте ошибок или падении worker health.
- **Check freshness** — можно вычислять по БД (максимальный `created_at` по `check_results` по стримам) или позже вынести в метрику.
- **Alert delivery** — пока опираться на логи и ручную проверку; при появлении счётчиков — формализовать.

Использование: эти SLO задают порог «нормальной» работы. Превышение (breach) — повод для разбора и приоритизации доработок, а не автоматическое наказание. Рекомендуется раз в квартал пересматривать целевые значения и окна.
