# Аудит проекта: разбиение на микросервисы

Документ фиксирует результаты глубокого аудита кодовой базы и рекомендации по выделению микросервисов: в первую очередь — **интеграция с AI** как отдельный сервис, а также другие независимые части системы.

---

## 1. Текущее состояние архитектуры

### 1.1 Компоненты и границы

| Компонент | Назначение | Зависимости |
|-----------|------------|-------------|
| **API** (cmd/api) | REST, auth/RBAC, tenant guard, CRUD, постановка check-jobs | Postgres, Redis (rate limit), локальный FS (скриншоты по путям) |
| **Worker** (cmd/worker) | Claim job → проверки (ffmpeg, playlist, segments, freeze, blackframe, bitrate) → persist → алерты → AI (опционально) → retention | Postgres, internal/ai, internal/telegram, APP_DATA_DIR |
| **Scheduler** (cmd/scheduler) | Периодически дергает API: компании → потоки → POST check-jobs | Только HTTP к API |
| **Init** | Миграции + опционально seed | Postgres |
| **Frontend** (web/) | Next.js, вызовы API | NEXT_PUBLIC_API_BASE_URL, INTERNAL_API_BASE_URL |

**Очередь задач:** не Redis, а **таблица `check_jobs` в PostgreSQL** (воркер забирает задачу через `ClaimNextQueuedJob` → `FOR UPDATE SKIP LOCKED`). Redis используется только для rate limiting в API.

**Данные:** одна БД PostgreSQL для API и Worker; скриншоты — локальный каталог `APP_DATA_DIR` (по умолчанию `/data`), пути хранятся в БД.

### 1.2 Домены в коде (уже модульно)

- **Тенанты:** companies, projects, streams (CRUD, tenant scope).
- **Проверки:** check_jobs, check_results — API ставит в очередь и читает результаты; Worker забирает, выполняет checks, пишет результаты.
- **Алерты:** alert_state, решение streak/cooldown/recovered, отправка в Telegram — только в Worker.
- **Инциденты:** открытие/закрытие, скриншоты, диагностика; API + Worker.
- **AI по инцидентам:** контракт B6 — только Worker вызывает `Analyzer`; результат сохраняется в `ai_incident_results`, API только отдаёт по GET.
- **Пользователи и регистрация:** auth, registration, approve/reject, Telegram-уведомление супер-админу.
- **Telegram:** доставка алертов (Worker), настройки по компании (API), DevLog (отдельный бинарник devnotify).
- **Retention:** только Worker — удаление старых check_results и файлов по TTL.

Граница **API ≠ Worker** соблюдается: тяжёлые операции и вызов AI только в Worker.

---

## 2. Цели разбиения

1. **AI-интеграция** — вынести в отдельный микросервис, чтобы подключать любые провайдеры (LLM, vision) без изменения Worker и с возможностью масштабирования и замены реализации.
2. **Дополнительные независимые части** — сервисы, которые логически отделены и могут развиваться/деплоиться отдельно: доставка уведомлений, тяжёлые проверки потоков, планировщик (уже вынесен).

---

## 3. Рекомендуемые микросервисы

### 3.1 AI-микросервис (приоритет 1)

**Назначение:** единственная точка вызова моделей для анализа инцидентов (cause/summary по метрикам + скриншоту). Соответствует контракту B6 (`docs/ai_incident_contract.md`).

**Текущее состояние:** Worker вызывает интерфейс `internal/ai.Analyzer`; реализация — `StubAnalyzer` (пустой ответ даже при `AI_INCIDENT_API_KEY`). Реальный провайдер планируется подключить позже.

**Предлагаемая схема:**

- **AI-сервис** — отдельный процесс (Go или другой стек):
  - Вход: HTTP/gRPC запрос с метриками проверки (checks), путём к скриншоту или его идентификатором (URL/object-id), контекст (company_id, stream_id, job_id).
  - Выход: `cause`, `summary` (как в `IncidentResult`).
  - Секреты (API keys провайдера) только в ENV AI-сервиса; в логах не отдавать.

- **Worker** перестаёт вызывать `internal/ai` напрямую и вместо этого:
  - Вызывает AI-сервис по HTTP (или кладёт задачу в очередь — см. ниже).
  - Получает ответ и сохраняет результат через существующий `ai_incident_repo` (как сейчас).

- **Варианты взаимодействия:**
  - **Синхронный HTTP:** Worker после сохранения check_result делает запрос к AI-сервису, получает cause/summary, сохраняет в `ai_incident_results`. Просто, но увеличивает время обработки одной job.
  - **Асинхронный (очередь):** Worker публикует сообщение «нужен анализ для job_id/check_result_id»; AI-сервис забирает, вызывает модель, пишет результат в БД или отдаёт в callback. Worker не блокируется; можно масштабировать AI-воркеры отдельно.

**Зависимости:**

- Worker → AI-сервис (HTTP или очередь).
- Общий доступ к скриншотам: либо общий volume (например `APP_DATA_DIR`), либо AI-сервис получает URL/идентификатор и сам читает из общего хранилища (в будущем — S3).
- Результаты AI: либо AI-сервис пишет в общую БД (`ai_incident_results`), либо возвращает данные Worker’у, и пишет Worker (как сейчас).

**Итог:** выделение AI в отдельный сервис даёт независимую интеграцию с любым провайдером, изоляцию секретов и возможность масштабирования и смены реализации без правок в Worker.

---

### 3.2 Сервис доставки уведомлений (Telegram / Alerts) — приоритет 2

**Назначение:** единая точка отправки алертов и, при желании, других уведомлений (например регистрация). Worker и API не дергают Telegram напрямую.

**Текущее состояние:** Worker сам читает `telegram_delivery_settings`, принимает решение об алерте (streak/cooldown) и вызывает `internal/telegram` (sendMessage). Уведомления о регистрации шлёт API через тот же пакет.

**Предлагаемая схема:**

- **Notification service** — отдельный процесс:
  - Получает задачи на отправку: из очереди (Redis/RabbitMQ) или по внутреннему HTTP от Worker/API.
  - Payload: тип (alert / registration / и т.д.), tenant (company_id), получатель (chat_id), текст, опционально метаданные.
  - Сервис хранит свои секреты (токены ботов), выполняет anti-spam и ретраи на своей стороне (или делегирует контракту очереди).

- **Worker:** вместо вызова Telegram кладёт в очередь сообщение «отправить alert для company_id, stream_id, текст, reason».
- **API:** при approve/registration кладёт в очередь «отправить уведомление супер-админу».

**Зависимости:**

- Worker, API → очередь или HTTP к Notification service.
- Notification service → Telegram API.
- Состояние anti-spam (cooldown, streak) сейчас в БД (`alert_state`); можно оставить в Worker (решение «слать/не слать») и в Notification только «доставка», либо перенести часть логики в сервис — по желанию.

**Итог:** изоляция всех внешних вызовов в Telegram в одном месте; замена транспорта (другой мессенджер, email) без изменений Worker/API.

---

### 3.3 Сервис проверок потоков (Stream Checker) — приоритет 3

**Назначение:** вынести тяжёлую часть Worker’а (ffmpeg, загрузка плейлистов, все checks) в отдельный масштабируемый сервис.

**Текущее состояние:** один Worker забирает job из БД, выполняет все checks (playlist, segments, freshness, freeze, blackframe, bitrate), пишет check_result и обновляет инциденты/алерты.

**Предлагаемая схема:**

- **Checker service** — один или несколько воркеров:
  - Забирает задачи из той же таблицы `check_jobs` (или из очереди) — как сейчас Worker.
  - Выполняет только «проверку»: скачивание, ffmpeg, расчёт метрик.
  - Пишет результат в БД (check_results, скриншоты в общий FS или S3) и переводит job в `done`/`failed`.

- **Текущий Worker** (или переименованный «Orchestrator» / «Alert Worker») остаётся для:
  - Обновления инцидентов (open/close, диагностика).
  - Решения об алерте (alert_state, streak/cooldown).
  - Вызова AI (или постановки задачи для AI-сервиса).
  - Постановки задачи в Notification service.
  - Retention cleanup.

Либо можно оставить один Worker, но вынести только «тяжёлые» checks в отдельный пул воркеров (общая БД, общий контракт job/result).

**Зависимости:**

- Общая БД (check_jobs, check_results) и общий объём для скриншотов (или общее хранилище).
- При разделении: Orchestrator → Checker по очереди или по БД (Checker только «выполняет проверку», Orchestrator — инциденты, алерты, AI, retention).

**Итог:** возможность масштабировать именно проверки (несколько инстансов Checker) без умножения инстансов алертов и retention.

---

### 3.4 Scheduler — уже микросервис

Scheduler уже вынесен: отдельный образ, только HTTP к API. При микросервисной архитектуре можно оставить как есть или заменить на общий cron-like оркестратор, который дергает API для постановки check-jobs.

## 4. Зависимости между сервисами (сводка)

| Сервис | Зависит от | Общие ресурсы |
|--------|------------|----------------|
| API | Postgres, Redis | БД, (опционально) общий FS для отдачи скриншотов |
| Worker (или Orchestrator) | Postgres, AI-сервис, Notification service | БД, общий FS/хранилище |
| AI-сервис | — (внешний провайдер в ENV) | Чтение скриншотов (volume/URL), запись в БД или ответ Worker’у |
| Notification service | Очередь/HTTP от Worker и API, Telegram | — |
| Checker (опционально) | Postgres (или очередь) | БД, общий FS/хранилище |
| Scheduler | API (HTTP) | — |

**Очереди:** сейчас только БД. При введении AI и Notification разумно добавить очередь (Redis streams, RabbitMQ и т.п.) для асинхронных задач и алертов.

**БД:** на первом этапе можно оставить одну PostgreSQL для API, Worker, AI-результатов и аудита; при росте — обсудить выделение БД по доменам.

## 5. Поэтапный план

1. **Фаза 1 — AI-микросервис**
   - Реализовать отдельный AI-сервис (HTTP API: вход — метрики + путь/ид скриншота, выход — cause/summary).
   - В Worker заменить вызов `internal/ai` на HTTP-клиент к AI-сервису (с таймаутом и без секретов в логах).
   - Результат по-прежнему сохранять в `ai_incident_results` (через Worker или напрямую из AI-сервиса — по решению).
   - Документировать контракт в `docs/ai_incident_contract.md` (адрес сервиса, формат запроса/ответа).

2. **Фаза 2 — Notification service (опционально)**
   - Ввести очередь сообщений (Redis/RabbitMQ).
   - Реализовать сервис доставки: потребление из очереди, отправка в Telegram.
   - Worker и API вместо прямого вызова `internal/telegram` публикуют сообщения в очередь.

3. **Фаза 3 — Checker (опционально)**
   - Вынести «проверку» (checks + ffmpeg) в отдельный образ/сервис с тем же контрактом job/result.
   - Оставить в текущем Worker только оркестрацию: инциденты, алерты, вызов AI, уведомления, retention.

4. **Общее**
   - Сохранять strict multi-tenant (company_id) во всех сервисах и очередях.
   - Секреты только в ENV/секретах контейнеров, не в логах.
   - Обновлять `docs/decisions.md` (ADR) при принятии решений по каждому шагу.
